## PDF가 어려운 진짜 이유

"PDF에서 텍스트 뽑는 게 뭐가 어렵냐"는 흔한 오해다. 실제 운영해보면 다음 다섯 가지가 동시에 터진다.

### 1. 텍스트가 이미지로 박혀 있다

스캔된 PDF, 워드에서 "그림으로 저장 후 PDF 변환"한 파일. 텍스트 추출 라이브러리로는 한 글자도 못 뽑는다. **OCR이 필요하다.**

### 2. 2단/3단 컬럼

논문, 법률 문서에 흔하다. 단순 텍스트 추출은 왼쪽 컬럼 1줄 → 오른쪽 컬럼 1줄 → 왼쪽 2줄 → ... 식으로 **읽는 순서가 꼬인다.** 레이아웃 인식이 필요하다.

### 3. 표 안의 셀 병합

업무 문서에서 가장 흔한 함정이다. 셀이 병합되거나, 행 안에 여러 줄이 들어가면 일반 라이브러리는 표 구조를 망가뜨린다. "표 1행 1열에 무슨 값이 있나"가 깨진다.

### 4. 도장, 서명, 체크박스

스캔 문서에 흔히 들어있다. 일반 OCR은 이걸 인식 못 하거나, 무의미한 문자로 변환한다. 계약서/공문에서는 결정적인 정보인데도.

### 5. 손글씨, 워터마크, 다국어 혼재

손글씨는 OCR 정확도가 급락한다. 워터마크가 본문과 겹치면 텍스트가 끊어진다. 한/영/일/중이 한 페이지에 섞이면 언어 감지가 흔들린다.

> 그래서 "Document AI"가 별도의 영역이 됐다. PDF 텍스트 추출은 1줄이지만, 운영 가능한 시스템은 1000줄이다.

> **Q.** PDF가 텍스트 기반인지 스캔 기반인지 어떻게 자동으로 판별하나? 파이프라인 분기가 필요한데.
>
> **첫 페이지에서 텍스트를 시험 추출해보고, 결과가 비어있으면 스캔으로 간주**한다. 한 줄짜리 휴리스틱이지만 신뢰도가 높다.
>
> 구현은 간단하다.
> ```python
> import fitz  # PyMuPDF
>
> doc = fitz.open("input.pdf")
> first_page_text = doc[0].get_text().strip()
> is_scanned = len(first_page_text) < 50  # 임계값은 도메인 맞춰 튜닝
> ```
>
> PyMuPDF의 `get_text()`는 PDF 안에 임베드된 텍스트 객체만 가져온다. 스캔 이미지에는 텍스트 객체가 없어서 빈 문자열이 나온다. 50자 미만 같은 임계값은 "헤더만 살아있고 본문이 스캔"인 케이스를 잡으려는 안전장치다.
>
> 좀 더 정교하게 가려면 **이미지 객체의 면적 비율**도 본다. 페이지 면적의 80% 이상이 이미지로 덮여있으면 스캔으로 판단.
>
> *판별 정확도보다 분기 안정성이 더 중요하다.* 잘못 판별돼도 OCR 파이프라인이 텍스트 PDF를 처리할 수 있고, 텍스트 파이프라인이 스캔을 만나면 빈 결과만 나오므로 큰 사고는 안 난다. 임계값에 너무 매달리지 말고 두 경로 다 안전하게 짠다.
{: .prompt-info }

## 도구 카탈로그

각 문제를 해결하는 도구들이다.

### OCR (이미지 → 텍스트)

| 도구 | 특징 | 추천 시나리오 |
|---|---|---|
| **Tesseract** | 전통적, 무료, 오픈소스 | 영문 중심, 간단한 문서 |
| **PaddleOCR** | 한국어/중국어 강함 | 한글 문서, 다국어 |
| **EasyOCR** | 빠른 진입 | 프로토타입 |
| **AWS Textract** | 클라우드, 표/양식 인식 | AWS 인프라 쓸 때 |
| **Azure Document Intelligence** | 양식 추출 강함 | 인보이스/송장 |
| **Google Document AI** | 영수증/신분증 특화 | 영수증 처리 |

> **OCR (Optical Character Recognition)**
>
> 이미지 안의 텍스트를 인식해 **기계 가독 문자 데이터로 변환**하는 기술. 1990년대부터 있던 영역이지만, 2020년 이후 딥러닝 기반(Tesseract 4+ LSTM, PaddleOCR)으로 정확도가 크게 올라갔다.
>
> VLM이 등장한 뒤 OCR의 위상이 미묘해졌다. VLM이 "OCR + 이해"를 한 번에 하기 때문이다. 하지만 OCR이 사라지진 않는다 — 대량 처리에서 OCR은 페이지당 ms 단위, VLM은 초 단위라 비용 차이가 100배 이상이다. **싼 OCR로 80%, 어려운 페이지만 VLM**이 표준 패턴이다.
{: .prompt-info }

### 통합 파싱 라이브러리

OCR + 레이아웃 분석 + 표 추출 + chunking까지 한 번에 해주는 도구들. 요즘 가장 인기다.

| 도구 | 특징 |
|---|---|
| **Unstructured.io** | 가장 인기, 모든 포맷 지원 (PDF/DOCX/HTML/이미지) |
| **Docling** (IBM) | 2024년 등장, 표 파싱 강함, 빠름 |
| **LlamaParse** | LlamaIndex 진영, 클라우드 SaaS |
| **PyMuPDF** | 저수준 PDF 파싱, 빠르고 가볍지만 noisy PDF에 약함 |

### 표 추출 특화

| 도구 | 특징 |
|---|---|
| **Camelot** | 격자형 표에 강함 |
| **Tabula** | Java 기반, 단순한 표 |
| **TableTransformer** (Microsoft) | 딥러닝 기반, 복잡한 표 |

### 레이아웃 분석

| 도구 | 특징 |
|---|---|
| **LayoutLMv3** | 텍스트+좌표+이미지 동시 입력, 분류 강함 |
| **DETR** (Detection Transformer) | 객체 영역(표/그림/제목) 검출 |
| **PaddleStructure** | PaddleOCR의 레이아웃 모듈 |

> **Q.** Unstructured와 Docling이 비슷해 보인다. 어떻게 골라야 하나?
>
> **다양한 포맷이면 Unstructured, 복잡한 표 중심이면 Docling**이다.
>
> 차이:
> - **Unstructured** — PDF/DOCX/HTML/이메일/이미지까지 20+ 포맷 지원. 출력 형식이 통일돼서 후처리 파이프라인이 깔끔. 다만 표 추출 정확도가 평범
> - **Docling** — IBM이 2024년 공개. PDF/이미지에 특화돼서 표 파싱 정확도가 매우 높음. 다만 지원 포맷이 좁음
>
> 실전 선택 기준:
> 1. **계약서·약관·정책** 위주 → Unstructured (포맷 다양, 표 적음)
> 2. **재무제표·송장·연구 논문** 위주 → Docling (표 많음)
> 3. **둘 다 섞임** → Unstructured를 기본으로, 표 추출이 깨지는 페이지만 Docling으로 처리
>
> *직접 평가셋으로 측정하는 게 가장 빠르다.* 도메인 문서 10장 정도를 두 도구로 돌려서 추출 결과를 비교하면 30분 안에 답이 나온다. 블로그 후기에 의존하지 말고 본인 데이터로 측정한다.
{: .prompt-tip }

## 현실의 워크플로우

도구 하나로는 안 된다. 보통 다음 파이프라인을 조립한다.

```text
[PDF 업로드]
   ↓
   ① 텍스트 PDF인지 스캔 PDF인지 판별 (PyMuPDF로 첫 페이지 확인)
   ↓
   ② 스캔이면 → OCR (PaddleOCR / AWS Textract)
      텍스트면 → 직접 추출
   ↓
   ③ 레이아웃 분석 (Unstructured / Docling)
      → 본문 / 표 / 그림 / 헤더 영역 분리
   ↓
   ④ 표는 별도 JSON으로 구조화
      그림/차트는 VLM에 보내 설명 추출
      본문은 chunking
   ↓
   ⑤ 텍스트 chunk + 표 데이터 + 그림 캡션 = 통합 인덱싱
   ↓
   ⑥ Vector DB에 저장 (chunk + metadata: 페이지, 영역 종류, 출처)
```

VLM 등장 이후 이 파이프라인이 단순해지긴 했다. "어려운 페이지는 그냥 VLM에 던지면 된다"는 옵션이 생겼기 때문이다. 다만 VLM은 비싸서 **싼 파이프라인으로 80%를 처리하고, 어려운 20%만 VLM에 보내는 하이브리드**가 현실적이다.

> **Q.** 새 양식의 문서가 추가되면 기존 파이프라인이 깨지기 쉽다. 회귀를 어떻게 막나?
>
> **라벨링된 회귀 평가셋 + CI 자동 실행**이 정석이다. 운영의 80%는 평가셋 관리다.
>
> 절차:
> 1. **양식별로 5~10장씩 샘플을 모은다** — 송장, 영수증, 계약서, 송장 v2 등
> 2. **각 샘플의 정답(추출 결과)을 사람이 라벨링한다** — JSON 형태로 저장
> 3. **추출 파이프라인 변경 시 자동 평가** — CI에서 평가셋을 돌려 각 필드별 정확도가 90% 이하로 떨어지면 PR 차단
> 4. **새 양식이 들어오면 평가셋에 추가** — 평가셋이 영원히 성장하는 자산이 됨
>
> 평가는 두 레벨로 나눈다.
> - **필드 단위** — 발주번호, 금액 같은 핵심 필드의 정확도 (정확 일치 또는 fuzzy match)
> - **문서 단위** — 한 문서 전체에서 추출된 필드 비율 (recall)
>
> *진짜 가치는 자신감이다.* 평가셋이 있으면 "이번 변경이 운영에 영향을 줄까?"라는 막연한 두려움이 사라진다. CI에서 95% 정확도가 유지되면 배포하고, 깨지면 막는다. 라벨링 비용이 크지만, 그 비용 없이는 신뢰할 만한 운영이 불가능하다.
{: .prompt-info }

## 백엔드 개발자가 부담 가질 영역

이 영역에서 진짜 어려운 건 모델이 아니라 **데이터 품질 관리**다.

- 같은 회사의 문서도 부서마다 양식이 다르다 → 양식별 파서 분기
- 새 양식이 추가되면 깨진다 → 회귀 테스트 데이터셋 운영
- 추출 정확도 측정 어렵다 → labeled 데이터로 evaluation pipeline 구축
- 한 페이지 처리 시간이 5~30초 → 대량 처리는 큐(SQS/Kafka)와 워커 풀

이건 ML이 아니라 **전형적인 데이터 엔지니어링과 백엔드 인프라**다. 백엔드 개발자의 강점이 그대로 쓰이는 영역이다.
