## 임베딩(Embedding)이란

**정의**: 텍스트(또는 이미지)를 **고정 길이의 숫자 벡터로 변환**하는 모델.

```text
"강아지" → [0.21, -0.43, 0.88, ..., 0.05]   (예: 1536차원)
"개"    → [0.22, -0.41, 0.87, ..., 0.04]
"자동차" → [-0.91, 0.33, -0.12, ..., 0.77]
```

핵심 속성은 단 하나다 — **의미가 비슷한 문장은 벡터 거리도 가깝다.** 거리 측정은 보통 cosine similarity로 한다.

> **Cosine Similarity**
>
> 두 벡터가 가리키는 방향이 얼마나 일치하는지 측정한다. 값의 범위는 `-1 ~ 1`이다. 1에 가까우면 같은 방향, 0이면 직교(무관), -1이면 정반대.
>
> 벡터의 **길이를 무시하고 방향만** 보기 때문에 문장 길이의 영향이 적다. 그래서 텍스트 임베딩에서 표준 거리 척도로 쓰인다.
{: .prompt-info }

백엔드 비유로 보면 이해가 빠르다.

- `SHA256` 해시: "같은가, 다른가"만 알려준다.
- **임베딩**: "얼마나 비슷한가"를 알려주는 해시.

이 한 줄이 임베딩의 전부다. "강아지"와 "개"를 시스템이 동의어로 인식하게 만드는 마법은 결국 **숫자 좌표 사이의 거리 계산**일 뿐이다.

### 대표 임베딩 모델

| 모델 | 종류 | 특징 |
|---|---|---|
| `text-embedding-3-small` | OpenAI API | 가장 무난, 호출만 하면 됨 |
| `BGE-M3` | 오픈소스 | 다국어, 한국어 잘됨, 셀프호스팅 |
| `multilingual-e5` | 오픈소스 | 가성비 좋음 |
| `CLIP` | 오픈소스 | 텍스트↔이미지 매칭 가능 |

선택 기준은 단순하다 — 외부로 데이터 보내도 되면 OpenAI API, 안 되면 BGE-M3 셀프호스팅.

> **Q.** 임베딩 차원이 클수록 정확도가 좋아지나? `text-embedding-3-small` 1536차원, `BGE-M3` 1024차원, `MiniLM` 384차원 — 무엇을 골라야 하나?
>
> 도메인에 맞는 **최소 차원**이 정답이다. 무조건 크다고 좋은 게 아니다.
>
> 비용 계산이 결정적이다. 1억 건 × 1536차원 × float32 = 약 **614GB**. 같은 데이터를 384차원으로 줄이면 **154GB**. 인덱스 메모리 사용량과 검색 지연이 차원에 비례해서 증가한다.
>
> 정확도 측면에서도 한계 효용이 빠르게 떨어진다. 1024차원과 1536차원의 검색 정확도 차이는 대부분의 실무 도메인에서 1~2%p 수준이지만, 비용은 1.5배다.
>
> *현실적인 결정 흐름은 이렇다.* 도메인 평가셋(질문 100개 + 정답 문서)을 만들고, 384/768/1024/1536 차원 모델로 각각 인덱싱해서 Recall@10을 측정한다. 정확도가 평탄해지는 지점이 적정 차원. 모르면 우선 1024차원 모델로 시작한다.
{: .prompt-info }

## Vector DB — 임베딩을 저장/검색하는 데이터베이스

임베딩을 만들어도 저장할 곳이 필요하다. 일반 RDBMS는 "벡터 거리 기준 정렬"을 빠르게 못 한다. 그래서 **Vector DB**가 등장했다.

정의는 간단하다 — 벡터를 저장하고 **"이 벡터와 가까운 top-K 벡터"** 를 빠르게 찾아주는 DB.

### 백엔드 관점에서의 비교

| DB 유형 | 쿼리 형태 | 매칭 방식 |
|---|---|---|
| 일반 DB (MySQL) | `WHERE id = 123` | 정확 일치 |
| 전문검색 DB (Elasticsearch) | `WHERE text MATCH '강아지'` | 키워드 매칭 |
| **Vector DB** | `WHERE vector ≈ [0.21, -0.43, ...]` | **의미 유사도** |

내부적으로는 HNSW, IVF 같은 ANN 알고리즘을 쓴다.

> **ANN (Approximate Nearest Neighbor)**
>
> 전수 비교(brute-force)는 N개 벡터마다 거리를 계산하므로 O(N). 데이터 1억 건이면 한 번의 검색에 수십 초가 걸린다.
>
> ANN은 인접 후보군만 빠르게 좁히는 방식으로 **정확도 95~99%**를 유지하면서 **속도를 수백~수천 배** 끌어올린다. **HNSW**(그래프 기반, Qdrant·Weaviate·pgvector)와 **IVF**(클러스터 기반, FAISS·Milvus)가 대표적이다.
{: .prompt-info }

### 대표 제품

| 제품 | 특징 | 추천 시나리오 |
|---|---|---|
| **Pinecone** | SaaS, 가장 쉬움, 비쌈 | 빠르게 PoC, 운영 부담 줄이고 싶을 때 |
| **Qdrant** | Rust 기반, 셀프호스팅, 가성비 | 비용 절감, 한국 데이터 외부 송출 금지 |
| **Weaviate** | GraphQL API, 하이브리드 검색 강점 | 키워드+벡터 동시 활용 |
| **pgvector** | Postgres extension | 기존 RDBMS에 얹기, 트랜잭션 필요 |
| **Milvus** | 대규모, 엔터프라이즈 | 수억 건 이상 벡터 |

> **Q.** 다섯 개 중 어떤 Vector DB를 골라야 하나? 결정 기준은?
>
> 의외로 단순하다. **이미 운영 중인 인프라**와 **벡터 규모**, 두 변수면 거의 결정된다.
>
> - 이미 Postgres 운영 중 + 벡터가 수백만 건 이하 → **pgvector**. 새 인프라 추가 없이 기존 트랜잭션·백업·권한 체계를 그대로 쓴다.
> - 신규 프로젝트 + 셀프호스팅 가능 + 외부 송출 금지 → **Qdrant**. Rust 기반이라 단일 노드 성능이 좋고 운영 부담이 낮다.
> - 빠른 PoC + 운영 인력 없음 + 비용 무관 → **Pinecone**. SaaS라 설치·튜닝이 전혀 없다.
> - 수억 건 이상의 초대규모 → **Milvus**. 분산 아키텍처로 수평 확장이 검증됐다.
>
> *진짜 중요한 건 한 번 고른 뒤 깊게 쓰는 것이다.* "Pinecone에서 Qdrant로 이관"이 PoC 다음 단계의 자연스러운 흐름인데, 임베딩과 메타데이터 스키마만 일치시키면 이관 비용 자체는 크지 않다. 처음부터 완벽한 선택보다 **빠르게 PoC를 돌리는 선택**이 비용 절감보다 우선이다.
{: .prompt-tip }

## pgvector — 5분 만에 시작하기

가장 진입장벽이 낮은 건 **pgvector**다. 기존 Postgres에 extension 하나 설치하면 끝난다.

```sql
-- 1. extension 활성화
CREATE EXTENSION vector;

-- 2. 테이블에 vector 컬럼 추가
CREATE TABLE documents (
  id SERIAL PRIMARY KEY,
  content TEXT,
  embedding VECTOR(1536)
);

-- 3. 유사도 검색 (<=> 는 cosine distance)
SELECT id, content
FROM documents
ORDER BY embedding <=> '[0.21, -0.43, ...]'::vector
LIMIT 5;
```

> **pgvector의 거리 연산자**
>
> 위 SQL의 `<=>`는 cosine distance를 의미한다. pgvector는 세 가지 거리 연산자를 제공한다.
>
> - `<=>` — cosine distance (방향 차이, 텍스트 임베딩 표준)
> - `<->` — L2 distance (유클리드 거리, 이미지 임베딩에 자주)
> - `<#>` — negative inner product (내적, 일부 모델 권장)
>
> 임베딩 모델 문서에서 권장 거리를 확인하고 그에 맞는 연산자를 써야 정확도가 보장된다. 권장 거리와 다른 연산자를 쓰면 검색 결과가 미묘하게 어긋난다.
{: .prompt-info }

백엔드 개발자에게 이 코드가 친숙하게 느껴진다면, 이미 임베딩+VectorDB 영역의 절반은 이해한 셈이다. **"새로운 데이터타입을 다루는 SQL"** 그 이상도 이하도 아니다.

> **Q.** 같은 문장을 OpenAI 임베딩으로 한 번, BGE-M3로 한 번 인덱싱했다. 둘을 같이 검색할 수 있나?
>
> 못한다. 서로 다른 벡터 공간이라 거리 계산이 무의미하다.
>
> 임베딩 모델은 각자 고유한 벡터 공간을 학습한다. OpenAI `text-embedding-3-small`의 "강아지" 벡터와 BGE-M3의 "강아지" 벡터는 좌표가 완전히 다르고, 두 벡터 사이의 cosine similarity는 무작위 노이즈에 가까운 값이 나온다.
>
> 따라서 **인덱스를 모델별로 분리**해야 한다. 한 컬렉션에 두 모델의 벡터를 섞으면 검색이 깨진다. 같은 의미인지 보려면 두 인덱스 각각에서 검색하고 결과를 합쳐야 한다.
>
> *운영에서 결정적인 함의는 모델 교체 비용이다.* 모델을 OpenAI에서 BGE-M3로 바꾸려면 **모든 문서를 재임베딩**해야 한다. 1억 건이면 API 호출 비용 + 시간이 만만치 않다. 그래서 처음 모델 선택이 중요하고, 모델을 자주 바꾸지 않을 구조로 설계해야 한다.
{: .prompt-info }
