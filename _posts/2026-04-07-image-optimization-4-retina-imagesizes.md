## "100픽셀이라며. 왜 256이 와요?"

지난 글 끝에 남겨둔 의문이 있었어요. `<Image width={100} />`이라고 적었는데, 네트워크 탭을 보면 실제로는 `?w=256` 같은 값이 찍혀 있더라고요. 처음엔 버그인 줄 알았어요. 그런데 코드를 따라가보니 Next.js의 의도된 동작이었어요.

## 레티나, 즉 DPR 이야기

요즘 노트북·폰은 대부분 **레티나 디스플레이**예요. Device Pixel Ratio(DPR)가 2 이상이라는 뜻이에요. CSS로는 100px이지만 실제 픽셀은 200px이라는 거예요.

만약 100×100 박스에 정확히 100px짜리 이미지를 넣으면? 레티나 사용자에겐 흐릿하게 보여요. 반면에 200px 이미지를 넣으면 선명해요.

`<Image>`는 이걸 알아요. 그래서 srcset에 자동으로 1x/2x를 같이 만들어줘요.

```html
srcset="
  /api/files/stream/123?w=128 1x,
  /api/files/stream/123?w=256 2x
"
```

여기서 128, 256이라는 숫자가 어디서 왔냐. **`next.config.mjs`의 `imageSizes` (또는 `deviceSizes`) 후보군** 중에서 가장 가까운 값을 골라온 거예요.

## `imageSizes` vs `deviceSizes`

문서를 읽으며 구분이 헷갈렸는데, 정리하면 이래요.

- **`deviceSizes`**: viewport 폭 단위. `sizes` prop을 쓸 때 활용돼요. 큰 이미지(히어로, 풀폭) 용도예요.
- **`imageSizes`**: 작은 이미지(아이콘, 썸네일, 아바타 등) 용도. `width`/`height`로 고정 크기를 줄 때 이 후보군에서 가장 가까운 값을 골라요.

우리 이미지의 99%는 썸네일이라서 `imageSizes` 쪽이 핵심이었어요.

기본값은 이래요.

```js
imageSizes: [16, 32, 48, 64, 96, 128, 256, 384]
```

100을 요청하면 1x는 128, 2x는 256이 잡혀요. 그래서 네트워크 탭에 256이 찍혔던 거예요. 의도한 결과긴 한데, 우리 입장에서는 이 후보군이 너무 많았어요. 캐시가 8가지로 갈리거든요.

## 디자이너분과의 토큰 합의

이쯤에서 디자이너분께 가서 여쭤봤어요. "썸네일 사이즈 종류가 디자인 시스템에 몇 가지 정도 있어요?" 같이 시안을 훑어봤더니 실제로 쓰이는 건 4가지 정도였어요.

| 토큰 | 용도 | CSS 폭 |
|---|---|---|
| `xsmall` | 채팅 사이드 미리보기, 카드 옆 작은 이미지 | 80px |
| `small` | 채팅/공지/출퇴근 보고서의 일반 썸네일 | 100px |
| `medium` | 갤러리 카드, 프로필 큰 사이즈 | 160px |
| `large` | 히어로/모달 미리보기 | 320px |

"이거면 충분해요"라는 답을 듣고 마음이 가벼워졌어요. 무한한 사이즈가 아니라 4개로 끝낼 수 있다면 캐시 전략이 훨씬 단순해져요.

## `imageSizes`를 다시 정의했어요

각 토큰의 1x/2x를 실제 픽셀로 풀어서 후보군을 구성했어요.

| 토큰 | CSS 폭 | 1x | 2x |
|---|---|---|---|
| xsmall | 80 | 80 | 160 |
| small | 100 | 100 | 200 |
| medium | 160 | 160 | 320 |
| large | 320 | 320 | 640 |

중복을 빼면 후보군은 `[80, 100, 160, 200, 320, 640]`. 8개에서 6개로 줄었고, 무엇보다 **모든 값이 디자인이 실제로 쓰는 값**이에요. 캐시에 들어간 어떤 파일도 "왜 있는지 모르는" 상태가 아니에요.

```js
// next.config.mjs
images: {
  loader: 'custom',
  loaderFile: './src/lib/image/streamFileLoader.ts',
  imageSizes: [80, 100, 160, 200, 320, 640],
  qualities: [75], // 우리는 q=75 한 가지로 통일
}
```

`qualities`도 같이 정리했어요. 기본값은 여러 quality 후보가 있는데, 우리에게 q는 한 종류면 충분했어요. 후보가 많을수록 같은 이미지를 여러 번 만들게 되니까요.

## 토큰의 의미를 다시 정의했어요 — "최소 요청 폭"

처음엔 `size="small"` = "100px 이미지를 받는다" 라고 머릿속에 있었어요. 근데 레티나 때문에 그게 정확하지 않더라고요. 그래서 팀에 한 줄로 약속을 적어놨어요.

> **size 토큰은 "보이는 CSS 폭"이고, 실제 요청 폭은 그 값 이상에서 imageSizes가 골라준다.**

이렇게 정의하면 "small이면 무조건 100px만 받는다"는 오해가 사라져요. small 토큰을 쓴 자리는 100px CSS 박스이고, 그 박스를 채우기에 충분한 (1x 100, 2x 200) 이미지를 받는다, 라고 일관되게 설명할 수 있어요.

## 검증

토큰 정리하고 나서 네트워크 탭을 다시 봤어요.

- 일반 모니터에서 채팅 썸네일: `?w=100`
- 레티나 노트북에서 같은 화면: `?w=200`
- 모달 큰 미리보기: `?w=640` (레티나 기준)

전부 우리가 정한 후보군 안에 있었어요. 캐시는 사이즈별로 독립적이라, 같은 이미지를 같은 토큰으로 다시 그리면 백엔드까지 안 가고 브라우저 캐시에서 끝나요.

## 마지막 퍼즐 — 이걸 코드에서 어떻게 강제하지?

후보군이 정해졌으니 이제 이 토큰을 **모든 컴포넌트가 일관되게 사용**해야 해요. 누군가 `<Image width={123} />` 이런 어정쩡한 값을 적어버리면 토큰의 의미가 사라지거든요.

그래서 `<Image>`를 그대로 노출하지 않고, **`size` 프롭만 받는 얇은 래퍼**를 만들기로 했어요. 다음 편이자 마지막 편에서 이 래퍼 디자인과, 마이그레이션 과정에서 부딪힌 사고들 — 채팅 이미지가 뜬금없이 거대해진 사건, `unoptimized` 함정 — 이야기를 풀어볼게요.
