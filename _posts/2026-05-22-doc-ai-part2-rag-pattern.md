## RAG는 왜 필요한가

LLM은 강력하지만 결정적인 약점이 있다.

- 학습 시점 이후 데이터를 모른다.
- 회사 내부 문서, 사내 정책, 최근 가격표 같은 비공개 정보를 모른다.
- 모르는 걸 "모른다"고 하지 않고 그럴듯하게 지어낸다(Hallucination).

**RAG (Retrieval-Augmented Generation)** 의 발상은 단순하다.

> "LLM이 모르는 정보를 미리 검색해서, 질문과 함께 LLM 프롬프트에 끼워넣어주자."

학습이 필요 없고, 운영 비용도 낮고, 새 문서가 추가되면 인덱싱만 다시 하면 된다. 그래서 2023년 이후 모든 엔터프라이즈 AI 시스템의 **사실상 표준 패턴**이 됐다.

> **Hallucination**
>
> LLM이 사실이 아닌 정보를 그럴듯하게 생성하는 현상. 모델이 학습 데이터에 없는 질문을 받았을 때 "모르겠습니다"라고 답하는 대신, 통계적으로 그럴듯한 문장을 만들어내는 특성에서 기인한다.
>
> 환각이 발생하는 근본 이유는 LLM이 "사실 검증"이 아니라 "다음 토큰 확률 예측"으로 동작하기 때문이다. 그래서 모델을 더 크게 키워도 0이 되지 않는다. RAG는 환각을 줄이는 가장 실용적인 수단이다 — 답변할 때 참고할 문서를 미리 제공하기 때문.
{: .prompt-info }

## 기본 흐름

```text
[사용자 질문]
   ↓
   ① 질문을 임베딩 → 벡터화
   ↓
   ② Vector DB에서 가장 비슷한 문서 top-5 검색
   ↓
   ③ 검색된 문서를 LLM 프롬프트에 첨부
      "다음 문서를 참고해서 답해줘: {문서1, 문서2, ...}
       질문: {원래 질문}"
   ↓
   ④ LLM이 답변 생성
```

핵심은 **②**다. "어떻게 좋은 문서를 잘 골라내느냐"가 RAG 시스템의 품질을 결정한다.

> **Q.** RAG와 파인튜닝, 언제 무엇을 골라야 하나?
>
> 도메인 지식 주입에는 **RAG가 거의 항상 정답**이다. 파인튜닝은 "응답 스타일이나 형식 학습"에만 쓴다.
>
> 이유는 비용과 갱신 주기다. 새 문서가 들어올 때마다 파인튜닝하면 GPU 시간이 매번 들어가고, 모델을 재배포해야 한다. RAG는 인덱싱만 다시 하면 끝 — 분 단위로 갱신 가능하다.
>
> 정확도 측면에서도 RAG가 유리하다. 파인튜닝된 모델은 학습한 문서를 "외운다"기보다 "비슷한 문장을 흉내 낸다"에 가까워서 사실 인용이 깨질 수 있다. RAG는 답변 시 원문을 컨텍스트로 직접 보기 때문에 인용 정확도가 높고, 출처도 같이 반환할 수 있다.
>
> *파인튜닝은 "이 회사의 정중한 톤으로 답해라" 같은 형식 학습에 쓴다.* 도메인 지식은 RAG, 답변 스타일은 파인튜닝 — 이 분리가 실전 표준이다.
{: .prompt-tip }

## 구체적 구현 단계

실제 코드 레벨에서 RAG는 6단계로 구성된다.

### 1. Chunking — 문서 쪼개기

긴 문서를 통째로 임베딩하면 검색 정확도가 떨어진다. 200페이지 PDF의 임베딩 하나로는 "3페이지에 있는 특정 문장"을 찾을 수 없다.

그래서 **500~1000 토큰 단위로 쪼갠다.** 의미 단위(문단/섹션)를 유지하는 것이 중요하다. 단어를 어중간하게 자르면 검색 품질이 깨진다.

흔한 전략:

- 고정 크기 (500 토큰씩) — 간단하지만 문맥 깨짐
- 의미 단위 (문단 경계) — 좋지만 구현 복잡
- 슬라이딩 윈도우 (오버랩 100 토큰) — 절충안, 많이 쓰임

> **Chunk Size의 트레이드오프**
>
> 청크가 작을수록 검색 정밀도(precision)는 높지만, 답변 생성에 필요한 컨텍스트가 잘려서 답이 부정확해진다. 청크가 클수록 답변에 충분한 정보가 들어가지만, 검색 점수가 노이즈에 묻혀 정밀도가 떨어진다.
>
> 실전 권장값은 **500~1000 토큰 + 오버랩 100 토큰**. 도메인이 매우 정형적(예: API 문서)이면 작게, 서술형이면 크게.
{: .prompt-info }

### 2. Embedding — 각 chunk 벡터화

지난 편에서 다룬 임베딩 모델로 chunk 하나당 벡터 하나를 만든다.

### 3. Indexing — Vector DB에 저장

벡터와 함께 **metadata도 같이 저장**한다. 출처(어떤 파일), 페이지 번호, 작성일 등. 나중에 답변에 인용(citation)을 붙이려면 필수다.

```python
vectorstore.add(
    embedding=[0.21, -0.43, ...],
    metadata={"source": "policy.pdf", "page": 23, "section": "환불"}
)
```

### 4. Retrieval — 유사도 검색

질문도 같은 임베딩 모델로 벡터화한 뒤, Vector DB에서 top-K(보통 5~20)를 가져온다.

### 5. Re-ranking — 다시 정렬

top-20을 그대로 LLM에 넘기면 컨텍스트가 너무 길어지고 노이즈도 많다. 그래서 **re-ranker 모델**로 top-20을 다시 정렬한 뒤 top-5만 LLM에 넘긴다.

대표 도구: `bge-reranker`, `Cohere Rerank API`.

선택사항이지만 품질을 크게 끌어올린다.

### 6. Generation — LLM 호출

```text
시스템 프롬프트:
"아래 컨텍스트만 사용해서 답하세요. 컨텍스트에 없으면 모른다고 답하세요."

컨텍스트:
[문서1] policy.pdf p.23: "환불은 7일 이내 가능합니다..."
[문서2] policy.pdf p.24: "단, 다음 경우는 예외..."

질문:
"환불 가능 기간이 어떻게 되나요?"
```

이 프롬프트를 LLM에 던지면, **컨텍스트 기반의 사실적 답변**이 나온다.

> **Q.** RAG 응답 품질이 낮다. 어디부터 점검해야 하나?
>
> **검색(Retrieval) 단계부터 본다.** 80%는 여기서 깨진다.
>
> 점검 순서는 거꾸로 거슬러 올라간다.
> 1. **사용자 질문에 대한 정답 문서가 인덱스에 있는가?** — 없으면 인덱싱 단계 문제. 청킹/임베딩/메타데이터 확인.
> 2. **top-K에 정답 문서가 들어오는가?** — 안 들어오면 검색 문제. 임베딩 모델 교체나 Hybrid Search 도입 검토.
> 3. **들어오긴 하지만 순위가 낮은가?** — Re-ranking 추가로 해결.
> 4. **top-K에 정답이 있는데도 LLM이 틀리게 답하는가?** — 그제서야 프롬프트 문제. 시스템 프롬프트나 컨텍스트 포맷 수정.
>
> 디버깅의 핵심은 **각 단계의 입출력을 따로 로깅**하는 것이다. Langfuse 같은 도구로 retrieval 결과와 LLM 입출력을 분리해서 보면 어디서 깨지는지 즉시 드러난다.
>
> *흔한 함정은 LLM부터 의심하는 것이다.* LLM을 더 큰 모델로 교체해도 검색이 잘못된 문서를 가져오면 답변은 좋아지지 않는다. 디버깅 시간의 80%는 retrieval 단계에 써야 한다.
{: .prompt-info }

## 고급 RAG 패턴

기본 RAG는 의외로 잘 안 된다. 질문이 모호하거나, 정답 문서가 의미적으로 멀리 있으면 검색이 실패한다. 그래서 실전에서는 다음 변형을 자주 쓴다.

### Hybrid Search

벡터 검색은 의미를 잡고, BM25 키워드 검색은 정확한 용어를 잡는다. **둘을 결합**하면 둘 다의 장점을 얻는다.

```text
final_score = α * vector_score + (1-α) * bm25_score
```

`α`는 보통 0.5~0.7. 도메인에 따라 튜닝한다.

### Multi-query (Query Expansion)

원본 질문 하나로 검색하면 누락이 많다. LLM에게 **"이 질문을 다른 표현으로 3개 만들어줘"** 라고 시킨 뒤, 각각 검색하고 결과를 합친다.

```text
원본: "환불 어떻게 해요?"
→ LLM이 변형:
   - "환불 절차"
   - "구매 취소 방법"
   - "결제 취소 정책"
→ 3개 다 검색 → 결과 합집합
```

### HyDE (Hypothetical Document Embeddings)

질문을 직접 임베딩하지 않고, **LLM이 먼저 가짜 답변을 생성**한 뒤 그 답변을 임베딩해서 검색한다. "질문 임베딩"보다 "답변 임베딩"이 정답 문서와 가깝다는 인사이트 기반이다.

### GraphRAG

문서들을 그래프(entity-relation)로 만들어두고, 검색 시 관련 노드를 따라 확장한다. 마이크로소프트가 2024년에 정식 도입했다. 도메인 지식이 복잡하게 얽힌 경우(법률, 의료) 효과적이다.

> **Q.** top-K 값을 5로 할지 20으로 할지 어떻게 정하나?
>
> **컨텍스트 윈도우 비용과 답변 품질의 함수**다. 도메인 평가셋으로 측정해서 정한다.
>
> 직관과 다르게 K가 크다고 항상 좋은 게 아니다. K=20을 LLM에 넘기면 컨텍스트가 길어져 토큰 비용이 4배가 되고, 노이즈 문서가 답변에 섞일 가능성이 늘어난다("길어진 컨텍스트 속에서 LLM이 핵심을 놓치는" 현상이 알려져 있다).
>
> 표준 절차는 이렇다.
> 1. 평가셋(질문 50~100개 + 정답 문서)을 만든다.
> 2. K=5, 10, 20으로 각각 돌려서 답변 정확도와 토큰 비용을 측정한다.
> 3. **정확도가 평탄해지는 지점**의 K를 채택한다. 보통 K=5~10에서 정확도가 거의 다 차고, 그 이상은 비용만 늘어난다.
>
> *Re-ranking을 도입하면 답이 더 명확해진다.* "검색 K=20 → Re-rank top-5"로 가는 게 정석. 검색은 넓게(recall), LLM에 넘기는 건 좁게(precision).
{: .prompt-tip }

## 백엔드 관점에서의 RAG

여기까지 정리하면 한 가지가 명확해진다. **RAG는 ML이 아니라 데이터 파이프라인이다.**

- 모델은 남이 만든 걸 호출하기만 한다.
- 본인이 짜는 코드의 90%는 **데이터 흐름, 저장, 검색, API 결합**이다.
- 즉, 백엔드 개발자가 익숙한 영역의 연장선이다.

새로 배워야 하는 건 "벡터"라는 데이터 타입과 "LLM"이라는 외부 API뿐이다. 그 외에는 어디서나 보던 ETL과 API 백엔드다.
